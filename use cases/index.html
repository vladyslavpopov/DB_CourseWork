# Сценарій CommentModeration

ID CommentModeration
НАЗВА Модерація коментарів у соціальній мережі
УЧАСНИКИ Адміністратор, Система
ПЕРЕДУМОВИ Користувач залишив коментар, адміністратор має права на модерацію
РЕЗУЛЬТАТ Коментар опубліковано або видалено залежно від рішення адміністратора
ВИКЛЮЧНІ СИТУАЦІЇ 1. Користувач залишив недопустимий коментар
2 Коментар вже видалено
ОСНОВНИЙ СЦЕНАРІЙ 1. Адміністратор переглядає нові коментарі в системі
2. Система перевіряє наявність недопустимого контенту
3. Адміністратор вибирає коментар для перегляду
4. Система надає опції для публікації або видалення коментаря
5. Адміністратор приймає рішення про публікацію або видалення
6.Система повідомляє користувача про результат (коментар опубліковано або видалено)

# Сценарій MediaCreate

ID MediaCreate
НАЗВА Створення медіа-контенту
УЧАСНИКИ Користувач, система
ПЕРЕДУМОВИ 1. Користувач має доступ до інформаційної системи.
2. Користувач зареєстрований в системі.
3. Інформаційна система готова до отримання нового контенту (інтерфейс, ресурси, тощо)
РЕЗУЛЬТАТ Новий медіа-контент успішно створено та збережено в системі. Користувач отримує підтвердження про успішне створення контенту та можливість перегляду та редагування.
ВИКЛЮЧНІ СИТУАЦІЇ 1. InsufficientPermissionsException — користувач намагається створити контент без необхідних прав доступу.
2. FileUploadException - помилка при завантаженні медіа-файлу (наприклад, недопустимий формат, або перевищено ліміт розміру.
3. ServiceUnavailableException — система тимчасово недоступна, через технічні роботи або збої.
ОСНОВНИЙ СЦЕНАРІЙ 1. Користувач відкриває систему.
2. Користувач вибирає спеціальну опцію "Створити медіа-контент".
3.Система відображає форму для введення деталей медіа-контенту, та завантаження медіа-файлу.
4. Користувач заповнює необхідні поля, завантажує медіа-файл та натискає кнопку "зберегти".
5. Система перевіряє дані. Якщо дані коректні, система зберігає контент і повертає повідомлення про успішне стаорення. Якщо дані некоректні, система повідомляє помилки та пропонує їх виправити.
6. Користувач може переглядати новий контент у своєму профілі, або продовжувати створення нового контенту.

# Сценарій MediaEdit

ID MediaEdit
НАЗВА Редагування медіа-контенту
УЧАСНИКИ Користувач, система
ПЕРЕДУМОВИ 1. Користувач має доступ до інформаційної системи.
2. Користувач зареєстрований в системі.
3. Медіа-контент вже створений та доступний для редагування.
4. Користувач має права на редагування контенту.
РЕЗУЛЬТАТ Медіа-контент успішно відредагований та збережений в системі. Користувач отримує підтвердження про успішне редагування контенту.
ВИКЛЮЧНІ СИТУАЦІЇ 1. InsufficientPermissionsException — користувач намагається редагувати контент без необхідних прав доступу.
2. FileUploadException — помилка при завантаженні нового медіа-файлу (недопустимий формат або перевищення ліміту розміру).
3. ContentNotFoundException — система не може знайти медіа-контент для редагування.
4. ServiceUnavailableException — система тимчасово недоступна через технічні роботи або збої.
ОСНОВНИЙ СЦЕНАРІЙ 1. Користувач відкриває систему.
2. Користувач вибирає опцію "Редагувати медіа-контент".
3. Система відображає форму з поточними деталями медіа-контенту.
4. Користувач вносить зміни до деталей контенту або завантажує новий медіа-файл.
5. Користувач натискає кнопку "зберегти".
6. Система перевіряє дані.
7. Якщо дані коректні, система зберігає відредагований контент і повертає повідомлення про успішне редагування.
8. Користувач отримує підтвердження та може переглядати відредагований контент.

# Сценарій MediaFind

ID MediaFind
НАЗВА Пошук медіа-контенту користувачем
УЧАСНИКИ Користувач, Система
ПЕРЕДУМОВИ 1. Користувач має доступ до інформаційної системи.
2. Користувач зареєстрований в системі.
3. База даних медіа-контенту актуальна і містить інформацію.
РЕЗУЛЬТАТ Користувач отримує релевантні результати медіа-контенту згідно з його запитом.
ВИКЛЮЧНІ СИТУАЦІЇ 1. NoAccessException — користувач не може здійснити пошук через проблеми з підключенням, або проблеми на стороні сервера.
2. ContentNotFoundException — запит користувача не дав жодних результатів. Користувач отримує повідомлення про те, що контент не знайдено.
3. InsufficientPermissionException — користувач намагається отримати доступ до контенту, який потребує спеціальних прав, яких не має користувач.
ОСНОВНИЙ СЦЕНАРІЙ 1. Користувач відкриває інформаційну систему.
2. Вводить ключові слова для пошуку медіа-контенту у відповідне поле.
3. Система обробляє запит і здійснює пошук в базі даних.
4. Система повертає список релевантних даних.
5. Користувач переглядає результати.
6. Користувач завершує сесію, або продовжує пошук.

# Сценарій MediaDelete

ID MediaDelete

УЧАСНИКИ Користувач, система
ПЕРЕДУМОВИ 1. Користувач має доступ до інформаційної системи.
2. Користувач зареєстрований в системі.
3. Медіа-контент створений і доступний для видалення.
4. Користувач має права на видалення контенту.
РЕЗУЛЬТАТ Медіа-контент успішно видалено з системи. Користувач отримує підтвердження про видалення.
ВИКЛЮЧНІ СИТУАЦІЇ 1. InsufficientPermissionsException — користувач намагається видалити контент без необхідних прав доступу.
2. ContentNotFoundException — система не може знайти контент для видалення.
3. ServiceUnavailableException — система тимчасово недоступна через технічні роботи або збої.
ОСНОВНИЙ СЦЕНАРІЙ 1. Користувач відкриває систему.
2. Користувач вибирає опцію "Видалити медіа-контент".
3. Система запитує підтвердження для видалення контенту.
4. Користувач підтверджує видалення.
5. Система видаляє контент.
6. Система повертає повідомлення про успішне видалення.
7. Користувач отримує підтвердження, що контент успішно видалено.

# Сценарій UserRegistration

ID UserRegistration
НАЗВА Реєстрація користувача в системі
УЧАСНИКИ Користувач, Система
ПЕРЕДУМОВИ 1. Користувач не зареєстрований у системі.
2. Користувач увійшов за допомогою уснуючого облікового запису.
РЕЗУЛЬТАТ Створення облікового запису
ВИКЛЮЧНІ СИТУАЦІЇ - 1. AlreadyRegistered — користувач вже зареєстрований 2. MissingData — відсутні необхідні реєстраційні дані 3. InvalidData — некоректні дані
ОСНОВНИЙ СЦЕНАРІЙ 1. Користувач відкриває сторінку реєстрації
2. Вводить особисті дані (ім'я, прізвище, email, пароль)
3. Система перевіряє введені дані
4. Створюється новий обліковий запис
5. Користувач отримує підтвердження про успішну реєстрацію

# Сценарій UserAuthorization

ID UserAuthorization
НАЗВА Авторизація користувача
УЧАСНИКИ Користувач, Система
ПЕРЕДУМОВИ Користувач вже зареєстрований у системі
РЕЗУЛЬТАТ Створення облікового запису
ВИКЛЮЧНІ СИТУАЦІЇ 1. InvalidCredentials — введено некоректні дані
2. NotRegistered — користувач не зареєстрований
3. TooManyAttempts — забагато спроб входу
ОСНОВНИЙ СЦЕНАРІЙ 1. Користувач відкриває сторінку входу
2. Вводить дані для авторизації
3. Система перевіряє обліковий запис
4. Перевіряються дані авторизації
5. Користувач отримує доступ до свого облікового запису

# Сценарій DeleteAccount

ID DeleteAccount
НАЗВА Видалення облікового запису
УЧАСНИКИ Адміністратор, Система
ПЕРЕДУМОВИ 1. Адміністратор має доступ до системи управління.
2. Адміністратор зареєстрований в системі.
3. Обліковий запис існує і доступний для видалення.
РЕЗУЛЬТАТ Обліковий запис успішно видалено з системи.
ВИКЛЮЧНІ СИТУАЦІЇ 1. NoAccessException — адміністратор не може здійснити видалення через проблеми з підключенням або проблеми на стороні сервера.
2. AccountNotFoundException — обліковий запис не знайдено, або він вже видалений.
ОСНОВНИЙ СЦЕНАРІЙ 1. Адміністратор відкриває панель управління.
2. Вибирає користувача, чий обліковий запис потрібно видалити.
3. Система обробляє запит і видаляє обліковий запис.
4. Система повідомляє про успішне видалення облікового запису.
5. Адміністратор завершує сесію.

# Сценарій PromoteUserRole

ID PromoteUserRole
НАЗВА Підвищення ролі користувача
УЧАСНИКИ Адміністратор, Система
ПЕРЕДУМОВИ 1. Адміністратор має доступ до системи управління.
2. Адміністратор зареєстрований в системі.
3. Користувач, чия роль має бути підвищена, існує в системі.
РЕЗУЛЬТАТ Роль користувача успішно змінена.
ВИКЛЮЧНІ СИТУАЦІЇ 1. NoAccessException — адміністратор не може підвищити роль через проблеми з доступом або проблеми на стороні сервера.
2. UserNotFoundException — користувача не знайдено.
ОСНОВНИЙ СЦЕНАРІЙ 1. Адміністратор відкриває панель управління.
2. Вибирає користувача, чию роль потрібно підвищити.
3. Адміністратор вибирає нову роль для користувача.
4. Система обробляє запит і змінює роль користувача.
5. Система повідомляє про успішне підвищення ролі.
6. Адміністратор завершує сесію.
